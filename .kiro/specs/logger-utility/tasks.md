# Implementation Plan

- [x] 1. Foundation: パッケージ雛形と共有型の定義
- [x] 1.1 パッケージ雛形と依存関係を整備する
  - `src/python_util/logging/` サブパッケージ（`__init__.py`, `types.py`, `config_loader.py`, `handlers.py`, `factory.py`）と `tests/logging/` の雛形ファイルを作成する
  - `pyproject.toml` の `requires-python` を `>=3.11` に設定し、`dependencies` に `rich` を追加する
  - `pdm install` が成功し、`python_util.logging` パッケージが空実装のままインポート可能であることを確認する
  - _Requirements: 1.3, 3.1_

- [x] 1.2 ロギング設定の値オブジェクトを定義する
  - `LoggingConfig`（`default_level`, `console_enabled`, `console_level`, `file_path`, `file_level`, `loggers`）と `LoggerOverride`（`file_path`, `level`, `console_level`）を不変（frozen）データクラスとして型ヒント付きで定義する
  - `Any`型を使用せず、全フィールドに明示的な型を付与する
  - `LoggingConfig()` のデフォルト値のみでインスタンス化でき、フィールドの型がすべて明示されていることをテストで確認する
  - _Requirements: 3.2, 3.3, 3.4, 4.1, 4.2, 5.1, 5.3_
  - _Depends: 1.1_

- [x] 2. Core: 設定読み込みとハンドラ構築
- [x] 2.1 (P) 呼び出し側pyproject.tomlの探索・解析を実装する
  - カレントワーキングディレクトリから親方向へ`pyproject.toml`を探索し、最初に見つかったファイルのみを採用する
  - `[tool.python_util.logging]`テーブルを`tomllib`で解析し、`level`/`file`/`console.enabled`/`console.level`/`file_level`/`loggers.<name>`を`LoggingConfig`へ変換する
  - `loggers.<name>`が複数階層で一致する場合は最も具体的な（最長の）一致を優先して解決する
  - テーブル不在、TOML構文エラー、未知のログレベル文字列の場合は`warnings.warn`で通知した上でデフォルトの`LoggingConfig`を返し、例外を送出しない
  - 正常な設定ファイル・テーブル不在・構文エラー・階層的なオーバーライド競合のそれぞれについて、期待した`LoggingConfig`または警告が得られることをテストで確認する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5, 4.1, 4.2_
  - _Boundary: Config Loader_
  - _Depends: 1.2_

- [x] 2.2 (P) コンソール用・ファイル用ハンドラの構築を実装する
  - `"%(message)s"`フォーマッタを持つ`rich.logging.RichHandler`を構築するコンソール用ハンドラ構築処理を実装する
  - 時刻・レベル・ロガー名・メッセージを含むプレーンな`Formatter`を持つ追記モードの`logging.FileHandler`を構築するファイル用ハンドラ構築処理を実装する
  - ファイル用ハンドラ構築時、出力先ディレクトリが存在しなければ作成し、作成に失敗した場合は`warnings.warn`で通知した上で`None`を返す（例外は送出しない）
  - コンソール用・ファイル用それぞれに独立したログレベルしきい値を設定できるようにする
  - 正常系でのハンドラ生成、存在しないディレクトリの自動作成、ディレクトリ作成失敗時に`None`が返り警告が発生することをテストで確認する
  - _Requirements: 1.3, 2.1, 2.2, 2.3, 5.1, 5.2, 5.3_
  - _Boundary: Handler Builder_
  - _Depends: 1.2_

- [x] 3. Integration: get_logger ファサードの統合
- [x] 3.1 get_logger の設定解決・ハンドラ接続を実装する
  - 名前省略時に呼び出し元モジュール名をデフォルト名として解決する
  - 設定をプロセス内で一度だけ読み込みキャッシュし、対象ロガー名に適用すべき出力先（グローバル設定 or `loggers`テーブルの個別上書き）を解決する
  - コンソール用・ファイル用ハンドラを構築して対象ロガーに接続し、`propagate = False`に設定してルートロガー経由の二重出力を防ぐ
  - `python_util.logging`パッケージから`get_logger`のみを公開APIとしてエクスポートする
  - `get_logger("app")`を呼び出すとコンソールへrich整形出力され、ファイル設定がある場合は同一メッセージが指定ファイルにも追記されることをテストで確認する
  - _Requirements: 1.1, 1.2, 1.4, 2.4, 4.1, 4.2_
  - _Depends: 2.1, 2.2_

- [x] 3.2 ハンドラ重複登録の防止とスレッドセーフなレジストリを実装する
  - 設定済みロガー名を記録するプロセス内レジストリを実装し、同名ロガーへの再呼び出しではハンドラを再登録せず既存の`logging.Logger`をそのまま返す
  - レジストリの判定・登録操作を`threading.Lock`で保護し、単一のcheck-and-set操作として扱う
  - 同名で`get_logger`を2回呼び出してもハンドラ数が増加しないこと、および複数スレッドから同時に同名の未設定ロガーを要求してもハンドラが1組しか登録されないことをテストで確認する
  - _Requirements: 4.3, 4.4_
  - _Depends: 3.1_

- [x] 4. Validation: 統合シナリオとフォールバック経路の検証
- [x] 4.1 複数モジュール構成での出力先統合・分離シナリオを検証する
  - 一時ディレクトリに`pyproject.toml`を用意し、複数のモジュール/クラス相当のロガーが単一ログファイルへ統合されるシナリオを検証する
  - 同じ一時プロジェクトで`loggers.<name>`による個別ファイル分離設定を行い、該当ロガーのみが指定ファイルに書き出されることを検証する
  - コンソール出力とファイル出力を同時に有効化した場合に、同一メッセージが両方の出力先に記録されることを確認する
  - 実際に書き出されたログファイルの内容とログレベルによるフィルタ結果が設定通りであることをテストで確認する
  - _Requirements: 2.1, 2.3, 2.4, 4.1, 4.2, 4.4_
  - _Depends: 3.2_

- [x] 4.2 設定不備・環境不備時のフォールバック経路を検証する
  - `pyproject.toml`が存在しない場合にデフォルト設定（コンソール出力のみ）で動作することを確認する
  - TOML構文エラーや未知のログレベル文字列を含む設定でも例外を送出せずデフォルトへフォールバックし、警告が発生することを確認する
  - ログ出力先ディレクトリの作成に失敗する環境（権限不足を模擬）でも、警告発行後にコンソール出力のみで動作が継続することを確認する
  - _Requirements: 2.2, 3.4, 3.5_
  - _Depends: 3.2_

- [x]* 4.3 ロガー再利用とログレベル境界の回帰テストを追加する
  - 同名ロガーの再取得で同一設定が維持されること、DEBUG〜CRITICALの各レベルで期待通りにメッセージがフィルタされることを網羅的に確認する
  - コンソール出力とファイル出力とで異なるログレベルが指定された場合に、それぞれ独立してフィルタされることを確認する
  - _Requirements: 1.4, 4.3, 5.1, 5.2, 5.3_
  - _Depends: 3.2_
